The idea in one line

From a sentence to a scheduled, tracked task

Someone types a request the way they would say it out loud. The tool understands it, lets them confirm the details, then makes it real — and keeps track of the response.

The moving parts, in plain terms

Three building blocks

The front desk

The web page. Where a person types the request and confirms the details. Nothing to install — it opens in a browser.

The back office

Supabase. The secure records room plus a few quick-acting clerks. It understands the request, keeps the master record, and decides what happens next.

The operations runner

n8n. The assistant that actually books Google Calendar, sends the email, and watches the inbox for replies.

Each piece has one clear job. That separation is what keeps the tool dependable and easy to change.

Where the data lives

One secure record per task, kept in the back office

Every task is stored in the back-office database (Supabase). One master record exists from the moment a task is created and is updated as things happen.

Each task record holds

  • Title, assignee name and email
  • Date, time and priority; any notes
  • A unique reference number
  • Status, the calendar link, and every reply

Why one record matters

  • A single source of truth — no copies drifting apart
  • The reference number ties the email, the reply and the task together
  • Nothing is duplicated or lost along the way

A deliberate design choice

Why the understanding step lives in the back office, not the automation

1

A person checks it first

Understanding happens up front, so the details can be reviewed and edited on screen — not buried inside the step that does the sending.

2

One master record

The back office owns the task and its reference number from the start; the AI’s reading is saved straight into that same record.

3

Right tool for each job

The back office is the brain and memory; the runner is the hands. Keeping thinking separate from doing makes both easier to trust and change.

4

The key stays locked away

The AI service key lives in the back office — never in the browser, and never spread across the automation tool.

The operations runner

n8n is the bridge to Google

n8n connects applications without custom code. It securely holds the Google sign-in and performs the real-world actions on the tool’s behalf.

What it does

  • Creates the calendar event and sends the invite (Accept / Maybe / Decline)
  • Sends the email to the assignee with reply instructions
  • Watches the shared inbox (about once a minute) for replies
  • Hands each cleaned reply back to the back office

Why this tool

  • Reliable, visual, and quick to change as needs evolve
  • No custom code to maintain for each connection
  • Keeps Google access in one controlled place

Important

This is an assessment build — not yet production-ready

It proves the full idea end to end. Before rolling it out across the organization, several essentials are needed. None are exotic — each closes a real, practical gap.

Access & identity

Make sure only the right people can use it and see the data.

Trustworthy data

Pick real employees with no ambiguity about who is meant.

Guardrails & operations

Prevent obvious mistakes and run it dependably day to day.

Guardrails and operations

Calendar conflict warning

Flag it upfront when a slot clashes with an existing meeting — for example, 4 PM is already booked.

No scheduling in the past

Block a 1 PM meeting when it is already 4 PM, so simple slips are caught before sending.

Private code & proper hosting

Keep the source code private and host the app on Vercel with a clean domain and dependable uptime.

Operational essentials

Activity / audit log, delivery and failure alerts, role-based permissions, and multi-time-zone support.
